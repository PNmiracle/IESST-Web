IESST-Web 每次 Push 后部署环境更新教程

适用于你现在的线上环境：GitHub 仓库 PNmiracle/IESST-Web，服务器目录 /opt/iesst-web，线上域名 https://iesst.xyz。

记住一句话：以后 GitHub 只用 main 作为线上部署分支。本地改完代码先推到 GitHub 的 main，然后登录阿里云 Workbench，让服务器强制同步 origin/main，最后重新构建 Docker。

最快更新命令

如果你确认代码已经 push 到 GitHub，并且这次只是普通功能或页面更新，在服务器直接执行这一组：

cd /opt/iesst-web
git fetch origin
git reset --hard origin/main
docker compose -f docker-compose.prod.yml up -d --build
docker compose -f docker-compose.prod.yml ps
curl http://127.0.0.1:8080/actuator/health
curl -i https://iesst.xyz/api/public/journals

最后一条应该看到 Content-Type: application/json，不能看到 <!doctype html>。

1. 本地电脑先提交并推送

在你本地项目目录执行：

git status
git add .
git commit -m "说明这次更新了什么"
git push origin main

现在建议只把 main 当作正式线上版本。其他开发分支可以保留在本地或临时使用，但线上服务器只跟随 GitHub 的 main。

2. 登录阿里云 Workbench

进入阿里云 ECS 控制台，找到服务器，然后：

ECS 控制台 -> 实例 -> 远程连接 -> Workbench

登录后进入项目目录：

cd /opt/iesst-web

3. 部署前确认当前版本

先看服务器当前分支和最近提交，方便出问题时回滚：

git status
git branch --show-current
git log --oneline -5

正常情况下应该在 main 分支，并且工作区干净。

4. 强制同步 GitHub 最新 main

git fetch origin
git reset --hard origin/main

这两条命令的意思是：服务器不自己开发代码，只把工作区重置成 GitHub 上的 main。这样可以避免 divergent branches 这类 pull 分叉错误。

这会丢弃服务器项目目录里没有提交到 GitHub 的代码改动。正常部署场景下这是对的，因为服务器只负责运行，不负责开发。

5. 重新构建并启动 Docker

docker compose -f docker-compose.prod.yml up -d --build

这会重新构建前端和后端镜像，并替换应用容器。正常不会删除 MySQL 数据。

6. 检查容器状态

docker compose -f docker-compose.prod.yml ps

理想状态是三个容器都 healthy：

iesst-app         Up ... (healthy)
iesst-mysql-prod  Up ... (healthy)
iesst-clamav      Up ... (healthy)

7. 检查后端健康

curl http://127.0.0.1:8080/actuator/health

应该返回：

{"status":"UP"}

如果失败，看应用日志：

docker logs --tail=120 iesst-app

8. 检查线上 API 是否正常

这个检查很重要，因为你之前遇到过 Nginx 返回 HTML 导致前端报 JSON 解析错误。

curl -i https://iesst.xyz/api/public/journals

正确结果应该包含：

Content-Type: application/json

如果看到下面这种，说明 Nginx 又把 API 请求转到了前端页面：

Content-Type: text/html
<!doctype html>

9. 检查网站页面

浏览器打开：

https://iesst.xyz
https://iesst.xyz/EI
https://iesst.xyz/admin

如果页面还是旧的，先强制刷新：

Mac：Command + Shift + R
Windows：Ctrl + F5

10. 如果需要无缓存重建

如果你确定 GitHub 已更新，但线上页面仍然像旧版本，可以执行无缓存构建：

cd /opt/iesst-web
docker compose -f docker-compose.prod.yml down
docker image rm iesst-academic-platform:latest
docker compose -f docker-compose.prod.yml build --no-cache
docker compose -f docker-compose.prod.yml up -d

然后重新验证：

docker compose -f docker-compose.prod.yml ps
curl http://127.0.0.1:8080/actuator/health
curl -i https://iesst.xyz/api/public/journals

11. 如果 Nginx API 又返回 HTML

先看启用的 Nginx 站点：

ls -l /etc/nginx/sites-enabled/

正常应只启用你的 iesst.xyz 配置。之前出错是因为同时启用了 iesst 和 certbot-temp。

sudo unlink /etc/nginx/sites-enabled/iesst
sudo unlink /etc/nginx/sites-enabled/certbot-temp
sudo nginx -t
sudo systemctl reload nginx

再测：

curl -i https://iesst.xyz/api/public/journals

12. 回滚到上一个版本

如果新版本有严重问题，先找到上一个可用 commit：

git log --oneline -10

然后切回那个 commit，例如：

git checkout a1b2c3d
docker compose -f docker-compose.prod.yml up -d --build

恢复到主分支继续更新：

git checkout main
git fetch origin
git reset --hard origin/main

13. 重要提醒

不要随便执行 docker compose -f docker-compose.prod.yml down -v。带 -v 会删除数据库 volume，线上数据会丢。只有全新部署、明确不要数据时才使用。
不要把服务器上的 .env 提交到 GitHub。里面有数据库密码、后台密码和 OSS AccessKey。
