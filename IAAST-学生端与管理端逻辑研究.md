# IAAST 学生端与管理端逻辑研究

## 1. 目标

本文档用于梳理 `https://www.iaast.cn/` 公开可见的学生登录、投稿后台、管理逻辑与数据库建模线索，并对照当前项目 `IESST-Academic-Platform` 的实现现状，给出下一步可落地的演进方案。

这不是对对方后台的“逆向还原”，而是基于公开页面、公开文案与当前项目状态做出的产品与技术设计推断。

## 2. 公开可见证据

### 2.1 首页公开行为

基于 2026-06-16 对 `https://www.iaast.cn/` 的公开访问，可直接看到以下特征：

1. 顶部存在用户入口：
   - `请登录`
   - `快速注册`
2. 首页存在浮动服务入口：
   - `在线咨询`
   - `关注我们`
   - `手机网站`
   - `常见问题`
   - `硕博合作`
   - `科研礼包`
3. 站点导航不仅有期刊，还包含会议、新闻、科研工具等模块。
4. SSR 输出中可见客户端状态字段：
   - `orderData`
   - `loginVisible`
   - `loginSuccessUrl`
   - `openId`
   - `binding:"wx"`

这些字段说明对方站点至少具备：

- 前台登录态
- 登录弹窗或登录显隐控制
- 登录后跳回原页面的逻辑
- 用户订单/投稿中心状态
- 微信绑定或第三方身份绑定能力

### 2.2 会议详情页公开行为

基于会议详情页 `https://www.iaast.cn/meet/home/Bx102meb` 的公开内容，可见：

1. 投稿须知中明确写到：
   - `所有论文须提交至投稿系统`
2. 注册费用说明中明确写到：
   - `发票可在投稿后台“我的发票”处填写对应信息进行申请`

这两个公开文案很关键，基本可以证明对方存在面向普通用户的“投稿后台/用户中心”，而且这个后台不仅看投稿，还至少包含：

- 投稿记录
- 投稿状态
- 发票申请
- 订单/注册费用关联

## 3. 当前项目现状

### 3.1 已有能力

当前项目已经具备：

- 官网前台
- 管理后台 `/admin`
- 管理员账号体系
- 学生登录页 `/student/login`
- 学生“我的订单”页 `/student/orders`
- 学生账号管理 `/admin/students`
- 投稿记录管理 `/admin/submissions`

### 3.2 真实限制

当前项目里的“学生订单中心”还只是 Demo 级别：

1. 学生账号表只有 `student_users`
2. 学生登录后查看的“订单”，实际来自 `submissions`
3. `StudentUserService.orders()` 本质是：
   - 取当前手机号
   - 用手机号去匹配 `submissions`
4. 也就是说：
   - 现在没有真正的订单表
   - 没有支付表
   - 没有发票表
   - 没有订单进度表
   - 没有附件表
   - 没有用户端消息/返修往来记录

### 3.3 当前实现与 iaast.cn 的差距

| 维度 | 当前项目 | iaast.cn 公开线索 |
|---|---|---|
| 学生登录 | 有 | 有 |
| 注册入口 | 仅前端入口感较弱 | 首页显式存在 |
| 登录后跳回 | 有基础能力 | 公开状态里明确有 `loginSuccessUrl` |
| 我的订单 | 伪订单，基于 submissions | 公开状态里有 `orderData` |
| 投稿后台 | 没有独立用户中心域模型 | 会议详情页明确写有投稿后台 |
| 发票申请 | 无 | 明确存在“我的发票” |
| 第三方绑定 | 无 | 公开状态里有 `openId` / `binding:"wx"` |
| 订单生命周期 | 无正式建模 | 极大概率存在 |

## 4. 推荐产品逻辑

### 4.1 角色划分

建议把系统分成 5 类角色：

1. 访客
   - 浏览首页、期刊、服务、关于我们
   - 提交咨询/投稿意向
2. 学生/作者用户
   - 注册、登录
   - 查看我的订单
   - 查看投稿进度
   - 补充稿件材料
   - 申请发票
3. 顾问/编辑
   - 跟进订单
   - 更新进度
   - 上传返修材料
   - 记录沟通
4. 财务/运营
   - 审核发票
   - 管理付款记录
   - 导出业务数据
5. 超级管理员
   - 管理账号、权限、内容、审计

### 4.2 学生端核心流程

建议学生端按下面的逻辑演进：

1. 访客提交表单
2. 系统生成一条“线索/咨询记录”
3. 顾问确认后转成正式订单
4. 订单绑定：
   - 学生账号
   - 服务类型
   - 目标期刊/会议
   - 当前阶段
5. 学生登录后在“我的订单”中看到：
   - 基本订单信息
   - 当前状态
   - 材料清单
   - 关键时间节点
   - 顾问留言
   - 发票申请状态

### 4.3 管理端核心流程

管理后台建议拆成：

1. 内容管理
   - 轮播图
   - 期刊
   - 服务套餐
   - 专家
2. 用户中心
   - 学生账号
   - 登录状态
   - 账号启停
3. 业务订单
   - 咨询线索
   - 正式订单
   - 订单阶段
   - 稿件/附件
4. 财务管理
   - 发票申请
   - 付款记录
5. 安全与审计
   - 管理员密码
   - 操作日志
   - 后续角色权限

## 5. 推荐数据库设计

### 5.1 建模原则

当前项目已经有 `student_users`，所以不建议直接推翻重来。更合适的做法是：

- 保留 `student_users` 作为前台用户登录主表
- 保留 `submissions` 作为最早的“线索/投稿意向表”
- 在其上新增真正的订单中心表

### 5.2 推荐表职责

#### `student_users`

继续作为前台账号表：

- 手机号登录
- 密码
- 显示名
- 启停状态

#### `submissions`

建议重新定位为：

- 官网线索表
- 初始投稿/咨询意向表
- 可以作为订单前置来源

不再把它直接等同于“我的订单”。

#### `student_orders`

正式订单主表，建议承载：

- 学生账号
- 来源 submission
- 订单号
- 服务类型
- 目标类型（SCI / EI / 会议 / 翻译润色 / 科学编辑）
- 标题
- 顾问
- 金额
- 支付状态
- 订单状态

#### `student_order_progress`

订单进度流水表，建议承载：

- 初评完成
- 期刊匹配中
- 已给出修改意见
- 等待作者补充材料
- 已投稿
- 审稿中
- 已录用
- 已完成

这个表是学生端“看进度”的关键。

#### `student_order_files`

订单附件表，建议承载：

- 初稿
- 修改稿
- 投稿模板
- 审稿意见
- 回复信
- 发票资料

#### `invoice_requests`

发票申请表，建议承载：

- 关联订单
- 发票抬头
- 税号
- 邮箱/地址
- 申请状态
- 开票金额

这张表直接对应 iaast.cn 公开页面里的“我的发票”。

#### `consultation_records`

咨询/沟通记录表，建议承载：

- 来源渠道
- 联系人
- 手机/邮箱
- 主题
- 顾问回复
- 跟进状态

这张表可以覆盖首页“在线咨询”和表单咨询。

## 6. 与当前项目的演进建议

### 6.1 第一阶段

先补齐数据库结构：

- `student_orders`
- `student_order_progress`
- `student_order_files`
- `invoice_requests`
- `consultation_records`

### 6.2 第二阶段

改造学生端：

- `/student/orders` 改为读取真正的订单表
- 新增订单详情页
- 新增“我的发票”

### 6.3 第三阶段

改造后台：

- 后台新增订单管理页
- 后台新增发票审核页
- 投稿线索可转正式订单

### 6.4 第四阶段

增强账号体系：

- 学生注册
- 短信验证码
- 微信绑定
- 更细粒度角色权限

## 7. 当前判断

结合公开证据与现有代码，可以得出比较稳的结论：

1. `iaast.cn` 的学生端绝不只是“登录后看一张简单列表”
2. 其用户中心更接近：
   - 投稿后台
   - 订单中心
   - 发票中心
3. 当前项目已经有不错的前后台骨架，但学生端业务域模型还明显偏浅
4. 下一步的关键，不是继续堆页面，而是先把“学生账号 -> 线索 -> 正式订单 -> 进度 -> 发票”的链路建完整
